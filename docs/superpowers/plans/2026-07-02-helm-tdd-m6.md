# helm-tdd M6 (Go Support) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Implementer routing:** route "Step: Implement" work to a fresh in-pool `implementer` subagent per task (**Sonnet 5**, per the validated M1-M5 house process). Task-level review is done by the ORCHESTRATOR directly, followed by one orchestrator whole-branch review before merge.

**Goal:** `runFeature` drives a **Go repo** end to end (map → baseline → scope → plan → RED/GREEN slice loop → verify → accept), selected via a new `language: "python" | "go"` input on the MCP surface, with `venvDir` becoming optional (required only for Python). Motivating consumer: the twiceshy repo (`go test -race ./...`, pure Go, Go 1.25).

**Architecture:** One seam, two implementations. A `TestToolchain` interface owns every language-coupled operation the FSM and gates perform today through `runPytest`/`runPytestVerbose`/venvDir: scoped test runs, whole-suite verbose runs with per-test outcomes, verify-ladder path runs, missing-symbol detection, RED-test linting, regression path units, and language-specific prompt wording. `pythonToolchain(venvDir)` wraps the existing functions **byte-identically** (pure refactor; the whole existing test suite must stay green unmodified in behavior). `goToolchain(workDir)` implements the same contract over `go test -json`. Capabilities the Go side doesn't support in v1 (mutation gate, refactor ratchet) are declared as flags on the toolchain and skipped by the FSM — documented residual gaps, not silent failures.

**Tech Stack:** Same as M0-M5. **ZERO new npm dependencies.** The Go toolchain (`go` on PATH, 1.22+) is an environment prerequisite exactly like the Python venv is today; a missing `go` binary must produce a clear error, not a hang. CI gains `actions/setup-go@v5`.

## Global Constraints

- Repo: `dotts-h/copilot-foundry` on GitHub, public. M0-M5 (PRs #1-#7) merged to `main`. Work on branch `helm-tdd-m6`, PR, orchestrator whole-branch review, then squash-merge — same flow as every prior milestone.
- **No new dependencies anywhere.** Go test output is parsed from `go test -json` (stdlib `JSON.parse` per line); no Go-side helper binaries are compiled or embedded.
- **Ground truths carried from M5 — build on these, do not re-litigate:** subscription OAuth works for the SDK; `bypassPermissions` needs `allowDangerouslySkipPermissions: true`; the leash is a WRITE leash with boundary-aware Bash matching; `evaluateLeash` confines writes to the workspace root; runs execute on isolated worktrees delivering `helm-tdd/<runId>` branches. None of that changes in M6.
- **Ground truths verified for M6 (2026-07-02, this machine) — do not re-verify:**
  1. `/tmp` on the brain is exec-able (`rw,nosuid,nodev` — no `noexec`), so `go test` binaries built under a `/tmp` worktree execute fine. Do NOT add GOTMPDIR plumbing.
  2. Go 1.25.11 is on PATH on the brain. GitHub `ubuntu-latest` needs `actions/setup-go@v5` added to `.github/workflows/ci.yml` (Task 2).
  3. `go test -json` (Go ≥1.24) emits per-test events `{"Action":"pass"|"fail"|"skip","Package":"<import path>","Test":"<TestName>"}` and build failures as `{"Action":"build-fail",...}` events plus `FAIL\t<pkg> [build failed]` output lines. Classify from **parsed events, not exit codes** — `go test` exit codes do not distinguish "tests failed" from "did not compile".
  4. A missing symbol in Go is a **compile error** (`undefined: <name>`), and it fails the *whole package* build. That is the canonical first RED for Go — the classifier must accept a build failure whose output matches `undefined: <functionName>` (both deterministic runs) as `failed_as_expected`, mirroring the PR #7 pytest collection-error semantics.
- **ScriptedBackend contract (M2 desync class):** the Python path's `backend.runPhase` call count is UNCHANGED. The Go path legitimately makes ONE FEWER call per slice (no REFACTOR phase — `supportsRefactor: false`); Go scripted tests must script for plan + RED + GREEN only. Do not conditionally add/remove any other call.
- **The git diff-guard, leash, worktree, and commit flow are language-independent and stay untouched.** `commitAll`'s `__pycache__` excludes are harmless for Go; leave them.
- **`hardenFsm.ts`, `fsm.ts` (M0), `packages/core`, and `constantMutantGate.ts`/`characterizationGate.ts` are OUT OF SCOPE** — they keep their direct pytest coupling. Only the `runFeature` pipeline is generalized.
- **Behavior-preservation gate for the refactor task (Task 3):** after threading the toolchain, every pre-existing test must pass with assertions unmodified (mechanical call-shape edits only, e.g. constructing `pythonToolchain(venvDir)` in test setup). If a test's *expected values* need to change, the refactor is wrong — stop and reconcile.
- No placeholders, no TODOs. `npm run typecheck && npm test` (in `apps/tdd/`) green after every task. Commit after every task.

### Task 2: goRunner — `go test -json` execution and parsing

**Files:**
- Create: `apps/tdd/src/goRunner.ts`
- Create: `apps/tdd/fixtures/go-add-kata/` (`go.mod` with `module go-add-kata` + `go 1.22`; `add_kata.go` with `package addkata` + `func Add(a, b int) int { panic("not implemented") }`)

- Test: `apps/tdd/test/goRunner.test.ts`

**Interfaces:**
- Produces (all exported from `goRunner.ts`):
  - `parseGoTestJson(raw: string): { tests: BaselineTestResult[]; buildFailed: boolean }` — pure parser. One JSON object per line (ignore unparseable lines); a test result is an event with `Test` set and `Action` in pass|fail|skip → `nodeId = `${Package}::${Test}``, outcome mapped pass→"passed", fail→"failed", skip→"skipped". `buildFailed` is true iff any event has `Action === "build-fail"` OR the raw output contains `[build failed]`.
  - `runGoTest(cwd: string, targetRelPath?: string): Promise<{ verdict: "passed" | "tests_failed" | "infra_error"; raw: string }>` — runs `go test -json <pkg>` where `<pkg>` is `./` + dirname(targetRelPath) + `/` when given (package containing the file; `./.` → `./`), else `./...`. Verdict: exit 0 → "passed"; parsed `buildFailed` → "infra_error"; any parsed test failure or nonzero exit without build failure → "tests_failed". `raw` = stdout + stderr. timeoutMs 300_000.
  - `runGoTestVerbose(cwd: string): Promise<{ tests: BaselineTestResult[]; raw: string }>` — `go test -json ./...`, returns parsed per-test results (empty array when no test files; a build-failed package contributes no test rows).
  - `goModulePath(cwd: string): Promise<string>` — reads `<cwd>/go.mod`, returns the `module <path>` value; clear error if missing.
  - `goPathUnit(modulePath: string, relPath: string): string` — pure: `dir === "." ? modulePath : `${modulePath}/${dir}`` for `dir = dirname(relPath)`.
  - `isGoMissingSymbolError(raw: string, functionName: string): boolean` — true iff raw matches `new RegExp(`undefined: (\\w+\\.)?${escapeRegExp(functionName)}\\b`)`.
- Reuses `BaselineTestResult`/`TestOutcome` types from `phases/baseline.ts` (import type only — no runtime coupling).

- [ ] **Step 1: Write the failing tests.** Parser tests with canned `go test -json` lines (pass, fail, skip, build-fail event, `[build failed]` output line, garbage lines ignored, no-test-files package). Pure-function tests for `goPathUnit` (root file → module path; `internal/x/y_test.go` → `module/internal/x`) and `isGoMissingSymbolError` (`undefined: Add` hits for functionName "Add"; `undefined: pkg.Add` hits; `undefined: Added` does NOT hit "Add" — word boundary). Integration tests against `fixtures/go-add-kata` using the REAL go toolchain (no test file → verbose returns `[]`; write a temp failing `add_kata_test.go` calling a missing symbol `Sub` → `runGoTest` verdict "infra_error" and `isGoMissingSymbolError(raw, "Sub")` true; write a real failing test on `Add` → "tests_failed"; make `Add` return `a + b` in a temp copy → "passed"). export interface TestToolchain {
  language: Language;
  runScoped(cwd: string, targetRelPath?: string): Promise<{ verdict: ScopedVerdict; raw: string }>;
  runOnPaths(cwd: string, paths: string[]): Promise<{ passed: boolean; raw: string }>;
  runVerbose(cwd: string): Promise<{ tests: BaselineTestResult[]; raw: string }>;
  pathUnit(relPath: string): string;                 // the unit runVerbose nodeIds are keyed by

  lintRedTest(source: string): RedLintResult;
  supportsMutationGate: boolean;
  supportsRefactor: boolean;
  buildRedPrompt(slice: PlannedSlice): string;
  planNouns: { repo: string; identifier: string };   // e.g. {repo: "Python", identifier: "Python function (a valid Python identifier)"}
}

export async function createToolchain(language: Language, venvDir: string | undefined, workDir: string): Promise<TestToolchain>;
