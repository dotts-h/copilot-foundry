# helm-tdd M6 — Multi-language runners (JS/TS + Go behind a TargetRunner seam)

**Goal:** helm-tdd covers the whole active stack: Python (existing), TypeScript/Svelte/React-Native
(vitest + jest), and Go — selected per target repo behind one `TargetRunner` seam.

**Why:** the brain's active repos are Python (marketdesk, chat/gateway), TS monorepos
(cosul, governor, wayfarer — vitest, pnpm), Svelte (chat/web — vitest), React Native
(wayfarer/mobile — jest), and Go (twiceshy). Today every gate, classifier, symbol extractor,
and mutation operator is hardwired to pytest + venv.

**Architecture:** extract a `TargetRunner` interface (`apps/tdd/src/runner/`) carrying every
language-specific behavior: env setup, test execution (quiet/verbose/paths), run classification
(passed / failed / harness_error), missing-symbol RED signatures, source/test file predicates,
symbol extraction, the 3-operator mutation gate, static gates, and the language-specific RED
prompt fragments. The FSM, gates, and phases depend only on the interface. Three
implementations:

- **PythonRunner** — existing behavior moved verbatim (pytest `-o addopts=`, venv python
  AST scripts). Zero behavior change is the Task 1 gate.
- **JsRunner** — vitest or jest (detected from the target package.json, walking up to the
  workspace root), binaries resolved via `node_modules/.bin` walk-up (package-manager
  agnostic execution), `ensureEnv` installs deps inside the run worktree (pnpm/npm/yarn/bun
  by lockfile), symbols via the TypeScript compiler API in-process, mutation via TS AST
  transforms, `tsc --noEmit` as a blocking static gate when present. `.svelte` files are
  fail-soft unparsed (the TDD unit loop targets `.ts` modules).
- **GoRunner** — package-scoped `go test`, compile-failure vs test-failure classification
  (`undefined: X` at compile time IS the canonical first RED for Go — no way to hide a
  missing symbol from the compiler), symbols and mutation via embedded Go scripts
  (`go/ast` + `go/printer`) run with the target's own toolchain — the same
  "target toolchain parses its own language" pattern as Python. `go vet` as a static gate.
  Mutants that fail to build are discarded (applied=false), never counted as kills.

Language resolution: explicit `language` in the run spec wins; else marker files in targetDir
(go.mod → go, package.json → js, pyproject/setup.py/requirements/pytest.ini → python).
`venvDir` becomes optional — required iff the resolved language is python. Monorepos are
handled by pointing `targetDir` at the package dir (e.g. `cosul/packages/core`).

**Mutation parity note:** the Python "constant" operator executes the target function to learn
the expected value; that is unsafe out-of-context for JS and impossible without compilation
for Go, so JS/Go ship the three syntactic operators (arithmetic-swap, comparison-swap,
boolean-negation). Score semantics unchanged: killed/attempted over applied operators;
zero applicable operators → score 1.

**Out of scope:** Playwright e2e (not the TDD unit loop), `.svelte` component symbol parsing,
mutation via external tools (Stryker/gremlins — revisit if the built-in operators prove too weak).

## Tasks

1. **TargetRunner seam** — pure refactor; PythonRunner implements the interface; FSM/gates/
   phases consume it. Gate: full `npm test` + `npm run typecheck` green with no assertion changes.
2. **JsRunner** — detection, worktree install, vitest/jest execution + verbose parsers
   (nodeId = `relPath::testName`), JS/TS missing-symbol signatures, TS-API symbols + mutation,
   tsc static gate. Unit tests on fixture outputs/sources.
3. **GoRunner** — go test scoping, build-vs-test classification, `undefined:` signatures,
   embedded go/ast symbol + mutation scripts, `testPathKey` = package dir (Go verbose output
   has no file paths), go vet static gate. Unit tests (Go toolchain present locally + in CI).
4. **Wiring + katas + surface** — resolveRunner auto-detect, venvDir optional, FSM
   ensureEnv/static-gates/prompt-fragment wiring, MCP schema + skills/tdd docs, kata-ts +
   kata-go fixtures with env-gated live E2E specs.

Validation: live kata E2E ×3 languages, then real-repo dogfoods (twiceshy Go slice, a cosul or
governor TS package slice) on the Cursor backend.

Implementation per RULE #0: all four tasks dispatched to Composer 2.5 via `ask-cursor` under
written specs; orchestrator gates + reviews each diff.
